## Vision

### 10x Check
The 10x version of rumb is not more verbs. It is the moment rumb stops being a place
you *put* work and becomes the thing that tells you what your work *means*. You come
back after a week, type one command, and it says: momentum moved to the MCP surface,
the auth question has resurfaced three times and never been decided, RUMB-0023 has
burned five failed runs with no status movement so it is spiraling, and these two
items look like duplicates. You queried nothing. The graph read itself back to you.

### Platonic Ideal (12 months)
Capture is frictionless, no "what is this?" freeze. The concept tree is plastic, so
promote/demote/merge are one gesture. Delivery is its own tree, so "what ships in v1"
is visible separately from "what the system is." History is replayable, so a model
can read how you got stuck and propose the exit. Every grooming move records what you
*rejected*, so the graph teaches the next person, or the next you. A solo builder
running this feels like a team with a great PM.

## Accepted Scope (this cycle)
- **Bet 1 (verbs):** `reparent`, `edit`, `recast`, `unlink`, `merge` in the shared
  core, CLI/MCP parity, on existing tables, written substrate-aware (complete events
  incl. title, claim-safe, single-transaction). Net-new primitive: row delete.
- **Bet 2 (history):** hybrid delta+intent substrate — `changesets`/`deltas` tables,
  per-helper before/after capture (no DB triggers, hand-threaded), one-time genesis
  snapshot, `undo`, read-only `at <ts>`.
- **E1 `rumb digest`:** deterministic computed summary over graph + event log
  (momentum by kind, stale/orphan items, spiral detection from failed-run counts +
  no status movement, recurring-title threads). No LLM.
- **E3 inbox + `capture`:** `init` seeds an inbox node; `rumb capture "text"` drops an
  ungroomed item there; grooming verbs move it out. digest surfaces stale inbox items.
- **E4 rejected-alternatives:** optional `--rejected`/`--why` on grooming verbs, stored
  in the changeset intent payload.
- **E5 capability tier:** `tier` field (routine/standard/hard) on items, set at create
  and via `recast`/`edit`, displayed (not filtered or dispatched) in `ready`/`digest`.
  Model dispatch and tier-based ordering are deferred with the frontier work (E6).

## Specifications locked this review
Pinned so implementation needs no scope re-asking (all defaults are tunable constants).
Updated for Codex cross-model corrections.
- **digest thresholds:** spiral = an item with >=N **consecutive** failed runs since
  its last meaningful status movement (default N=3), not lifetime failures + stale
  status; thread = >=2 items whose titles match after case + whitespace normalization
  (literal recurrence only, not semantic — the vision prose about "duplicates" / "the
  auth question" is aspirational, build to this wording); stale inbox = an inbox child
  with no event in >7 days; momentum = done-count by **kind-at-time-of-done** over the
  last 7 days (so `recast` does not retroactively rewrite history; record kind in the
  done event/changeset).
- **inbox node:** a normal numeric `RUMB-NNNN` item with kind `inbox`, seeded as a
  direct child of root. Its id is stored in a `meta` row (key `inbox_id`) so it is
  found by lookup, NOT by a non-numeric reserved id (the existing `normalize_item_id`
  only resolves numeric `RUMB-NNNN`). Captured children get kind `note`.
- **existing-repo seeding:** the inbox is created by a **migration**, not only by
  `init` — rumb's own `.rumb` already exists, so an init-only seed would leave capture
  with no parent. The migration creates the inbox and writes `meta.inbox_id`.
- **reserved-node guards:** root and the inbox cannot be reparented, merged,
  unlinked-from-parent, recast, claimed, or deleted; verbs reject operations targeting
  them with a clear error.
- **capture template:** `rumb capture "<text>"` creates an item under the inbox with
  kind `note`, status `draft` (raw captures never appear in `ready`), tier `standard`.
  Title = the text **normalized to a single line** (tabs/newlines → spaces, trimmed,
  truncated ~120 chars) to keep tree/CLI output clean; full original text stored in a
  new nullable `body` column on items (added in the tier migration).
- **tier + body columns:** new nullable columns; `tier` default `standard` (root and
  inbox seeded `standard`), `body` null except for captures. One schema migration (4+).
- **undo:** refuses any changeset that touched a claim AND runs a causality check —
  it must not undo a mutation that later claims/branches/worktrees/runs/proposals
  depend on (not just "claim touched"). Grooming verbs stay claim-safe (block
  destructive grooming on items with an active, unexpired claim).
- **row-delete referential policy:** `unlink` (edge delete) and `undo`-of-create
  (item delete) are the only deletes; both must define handling of referencing rows.
  Reserved nodes are never deletable. `merge` supersedes (sets status), never deletes
  the merged item, so its references stay valid history.
- **history is the one canonical substrate:** built first (see Sequencing); digest and
  replay read it, no parallel legacy-events path.
- **snapshot/replay scope:** snapshot serializes the grooming-relevant tables (items
  incl. tier/body, edges, meta) as JSON in a `snapshots` row; on-demand + one-time
  genesis snapshot (no auto cadence this cycle). Claims/proposals/runs are lifecycle
  state, out of grooming-undo/replay scope (consistent with undo refusing
  claim-touching changesets). Replay/`at` is keyed on `seq`/changeset-id, not timestamp.
- **CLI/MCP parity cost (effort realism):** each verb is ~7-8 surfaces — clap args,
  main dispatch, core API, MCP tool schema, MCP serializer, smoke test, docs, error
  mapping — not just a core method. Per-verb effort estimates above price the core
  method only; multiply for the full surface when scheduling.

## Teaching reach (Codex cross-model C2)
The "history teaches" value this cycle is **single-machine**: the next *you* weeks
later, and a model on this machine reading how a task got stuck. That is the primary
use case and needs no sharing. Cross-person teaching requires committed/exportable
history; `.rumb` is gitignored local runtime state today. E4 (rejected-alts) stays —
it is cheap and valuable solo. **TODO:** history export / committable state to unlock

## Sequencing (revised HISTORY-FIRST, per Codex cross-model review C1)
1. **Central mutation / change-log layer FIRST.** One internal mutation API that every
   write routes through, capturing `(table, pk_json, before_json, after_json)` into
   `changesets`/`deltas` in the same transaction, plus the one-time genesis snapshot.
   This is also where core.rs's mutation paths get decomposed (F1 happens HERE, not
   deferred). One canonical event substrate, no legacy-events vs changesets split.
2. **The 5 grooming verbs**, routed through the mutation API so they are captured from
   the first move. The row-delete primitive lands here WITH an explicit referential
   policy (what happens to claims/proposals/runs/events referencing a deleted row).
3. **E5 tier field + E3 inbox/capture**, riding the mutation API.
4. **E1 `rumb digest`** reading the one canonical substrate.
5. **E4 rejected-alts** (intent payload), **`undo`** (causality checks, not just
   "claim touched"), **`at <seq>`** (keyed on seq / changeset-id, NOT Unix-second ts).
6. **Dogfood:** groom rumb's own graph via MCP; capture + digest on the real graph.

Rationale (revised): Codex showed verbs-first leaves early grooming outside the
replayable history, lands the genesis snapshot too late, and briefly runs two event
systems. Building the central mutation layer first makes the teaching history complete
from move #1 and forces the core.rs decomposition up front where it is cheapest.

## Eng review decisions (2026-05-21, /plan-eng-review + Codex outside voice)
Locked architecture. Supersedes the looser sequencing above.

- **Staging (5 PRs, history-first):**
  - **PR0:** pure module split of core.rs — refactor only, existing tests green, zero
    behavior change.
  - **PR1:** `Mutation` recorder + changeset/delta/snapshot schema (unified timeline).
  - **PR2:** the 5 grooming verbs (reparent/edit/recast/unlink/merge) + drop
    `Status::Claimed`.
  - **PR3:** tier + body columns, inbox node + `meta.inbox_id`, capture.
  - **PR4:** digest + undo + `at <seq>` + rejected-alts.
- **Mutation API (D2):** a transaction-scoped `Mutation<'conn>` that OWNS the
  `Transaction` (it derefs to `Connection`; the existing `&Connection` helpers become
  its methods). Never expose both `&Transaction` and `&mut Mutation` to a caller
  (borrow fights). Each write captures before/after into a delta; on commit it writes
  changeset + deltas atomically with the data.
- **One unified timeline (C1):** ALL item/edge mutations route through the recorder,
  including the claim/run/proposal lifecycle, on a single `changesets.seq`. Grooming
  changesets are `undoable=true`; lifecycle changesets `undoable=false`. No parallel
  legacy-events timeline for state. (Codex: two timelines break `at<seq>`/undo/digest.)
- **Drop `Status::Claimed` (C1):** stop writing `items.status` for claims; an active
  claim becomes a read-time availability filter (`ready_items` already excludes
  active-claim rows, core.rs:638). Item status stays draft/ready/in_review/done/
  superseded/abandoned. Requires: update `doc/concepts.md` lifecycle section; a
  migration to settle existing `Claimed` items back to `ready`; reserve/release/fail
  stop touching `items.status`.
- **Undo rule:** refuse undo of a changeset if any LATER changeset (including
  non-undoable lifecycle ones) references the touched item, its descendants, or the
  touched edge endpoints; also refuse claim-touching changesets.
- **PR1 schema from day one:** `changesets(seq PK = global timeline, ts, actor, verb,
  intent_json, undoable, kind)`; `deltas(changeset_seq, delta_idx, table_name,
  pk_json, before_json, after_json)` — `pk_json` handles composite PKs (edges);
  `snapshots(seq, format_version, payload_json)`. Genesis snapshot is
  format-versioned so PR3's tier/body/meta columns bump the format cleanly.
- **Module boundaries (D3):** `model` (pure structs/enums), `store`
  (conn/migrations/queries/row-mapping), `history` (Mutation/changesets/deltas/
  snapshots), `graph` (parent/edge/dependency rules), `claims` (lifecycle + git),
  `grooming` (verbs; depends on history/store/graph), `facade` (`RumbProject`, kept as
  the public API that CLI + MCP consume). `history` must NOT depend on
  grooming/capture/digest.

## NOT in scope
- Delivery tree, parallel frontier, ordinal edges, split, Git resolution bridge,
  GTD product, model dispatch — all deferred (see above).
- promote/demote verbs — skipped; reparent suffices.
- Any LLM call inside rumb — rumb stays deterministic.

## Dream State Delta
This cycle moves rumb from an append-only, status-only graph to a reshapeable graph
with frictionless capture, a deterministic "where are we going?" view, a replayable
teaching history, and a work-weight signal. That is roughly half the platonic ideal.
The remaining half is the delivery tree, parallelism surfacing, and the Git bridge —
all deferred but unblocked by this substrate.
